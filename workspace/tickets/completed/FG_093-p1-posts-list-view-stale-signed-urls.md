# posts/getByUsername list query does not return stale Convex signed URLs in inline images

## Context

ce:review (`feature-add-editor`, 2026-05-02) Finding #3, api-contract reviewer at confidence 0.85. Concrete failure path:

1. User uploads an inline image via paste. Editor stores `{ src: "<Convex signed URL>", storageId: "kg2..." }` in the body.
2. User saves the post (`articles.mutations.update` writes the body verbatim). The signed URL was generated by the upload-URL flow and has a ~1-hour TTL.
3. After 1+ hour, another user visits `/[username]/posts`. `posts/getByUsername` (`packages/convex/convex/posts/queries.ts:19`) returns each post with `body` included via `postSummaryReturnValidator` (`packages/convex/convex/posts/helpers.ts:14` — `body: v.any()`).
4. `apps/mirror/features/posts/components/post-list-item.tsx:52` renders `<ContentBody content={post.body} />`. `packages/features/editor/components/content-body.tsx:31-38` renders `<img src={node.attrs?.src} />` — the original (now-expired) URL.
5. Browser shows broken image.

`getBySlug` correctly rewrites src on every read via `mapInlineImages(...)` (FR-05). The list query was missed because FR-05 names only "detail queries (getBySlug for both surfaces)" and the implementer followed the spec literally. The user-facing promise is broader.

The articles surface has `getByUsername` returning lighter shape — verify whether articles' equivalent list query is affected too, and either include or scope this ticket.

## Goal

After this ticket, no list-view page renders a broken image because of a stale signed URL. Image src values served by `posts/getByUsername` (and any equivalent articles list query) either come from a fresh `ctx.storage.getUrl(storageId)` on every read, or are not returned at all.

## Scope

- `packages/convex/convex/posts/queries.ts` `getByUsername` — apply `mapInlineImages(body, attrs => ...)` rewrite OR strip body from the validator.
- Mirror review for the articles equivalent list query (if any).
- Tests confirming the new contract.

## Out of Scope

- Backfilling old posts to use `storageId`-only refs — the rewrite is read-time, not stored.
- Adding pagination cost guarding — the list query is bounded by the existing `take(N)` limits.
- Revisiting the post-list-item rendering layer — it already reads through `ContentBody`, which is unchanged.

## Approach

Two policies — pick one:

**Policy A (recommended): rewrite in list query.** Loops over posts in `getByUsername`, calls `await mapInlineImages(post.body, async (attrs) => ({ ...attrs, src: await ctx.storage.getUrl(attrs.storageId) ?? attrs.src }))`. Cost: N image-node `storage.getUrl` calls per page. Bounded by typical list size and image-per-post count.

**Policy B: drop body from list shape.** Modify `postSummaryReturnValidator` to omit body, expose only excerpt/title/coverImage. List view renders excerpt-only; user sees images only when navigating to detail. Smaller payload, no read-time cost, but a UX change.

Recommend A unless list-view rendering of inline images was a deliberate UX call. Confirm with product owner before implementation.

- **Effort:** Small (Policy A) or Medium (Policy B + post-list-item refactor)
- **Risk:** Low (A) or Medium (B — UX change)

## Implementation Steps

1. Confirm policy choice (A or B) with one-line clarification — e.g., "list view should/shouldn't render full inline image bodies." Default to A.
2. Policy A: in `packages/convex/convex/posts/queries.ts` `getByUsername` handler, after fetching posts, await an `await Promise.all(posts.map(async post => ({ ...post, body: await mapInlineImagesWithStorage(ctx, post.body) })))`. Define the async helper inline or in `content/body-walk.ts` if it can stay pure (it can't if it calls ctx — keep it in queries.ts or a queries-helper).
3. Policy B: edit `packages/convex/convex/posts/helpers.ts` `postSummaryReturnValidator` to omit body. Update `apps/mirror/features/posts/components/post-list-item.tsx` to render an excerpt instead of `ContentBody`.
4. Add Vitest: post with inline image returned by `getByUsername` has fresh src (Policy A) or no body field (Policy B).
5. Mirror the policy decision for any equivalent articles list query (`getByUsername` in articles, if it exists with the same shape).
6. Run `pnpm --filter=@feel-good/convex test` and `pnpm --filter=@feel-good/mirror build`.

## Constraints

- The `getByUsernameForConversation` exemption (articles only, per FR-05) must be preserved — no body rewriting for AI-clone consumption.
- Whichever policy wins, document it in the spec or in a code comment so the next review of FR-05 doesn't repeat this gap.
- Cost: Policy A adds N storage.getUrl calls per list-page request — acceptable for personal-blog scale.
