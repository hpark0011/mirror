# Bio Tab — Spec

**Created:** 2026-04-30

## What the user gets

- A new **Bio** tab on every Mirror profile (`/@username/bio`) that lists the person's work and education history, oldest-first to newest, in a clean card stack.
- As the profile owner, I can add, edit, and delete entries from a simple form; I pick a kind (Work or Education), type a title, choose a start month/year, optionally an end month/year (or "Present"), and optionally a short description and a link.
- As any visitor, I can view the bio with no login required — same data the owner sees, minus the edit controls.
- When I chat with someone's clone, the agent already _knows_ their background — if I ask "where did you go to school?" or "tell me about your time at Acme", the answer pulls from their bio without them having to repeat it in the chat persona.
- I never see a broken state: empty bio shows a friendly empty card; saving an entry updates the list and refreshes the agent's knowledge in the same beat.

## Non-goals

- **No rich-text or image attachments per entry.** Description is a short plain-text string. We're not building a mini blog post editor here.
- **No manual drag-to-reorder.** Order is auto-derived from `startDate` desc. Manual ordering adds a position column, drag interaction, and migration risk for zero proven value at this corpus size.
- **No work/education-specific subfields (organization, degree, school).** A single freeform `title` field per the user's locked decision (option A in clarifications). Splitting into typed subfields can come later if needed.
- **No public visibility toggle per entry.** All entries are public for v1. A future "private notes" mode is a separate spec.
- **No auto-import from LinkedIn or other sources.** Manual entry only.
- **No retroactive backfill of bio data into RAG for users who already have entries** — there are none yet (table doesn't exist), so backfill is a no-op. Reusing the existing `backfillEmbeddings` action covers any future need.

### 2. How data flows

**Read (public):**
1. Visitor navigates to `/@username/bio`.
2. Next rewrites `/@username/bio` → `/[username]/bio`. The canonical `bio/page.tsx` returns `null`; the parallel slot `@content/bio/layout.tsx` runs first.
3. `@content/bio/layout.tsx` (server component) calls `preloadAuthOptionalQuery(api.bio.queries.getByUsername, { username })` using the `username` from `params` directly — the query resolves `username → userId` server-side via the `by_username` index. The preloaded query result is passed into the page via a context provider — same pattern as `apps/mirror/app/[username]/@content/articles/layout.tsx`, which uses `getByUsername` exactly this way (issues I8, iter2-Finding3).
4. `@content/bio/page.tsx` renders `<BioPanel>`, which consumes the preloaded query via `usePreloadedQuery`. **Hydration is seamless** — the client starts with the server's data, no loading flash.
5. Convex runs the query: `db.query("bioEntries").withIndex("by_userId", q => q.eq("userId", userId)).collect()` → sort desc by `startDate` then `_creationTime` desc → `slice(0, 50)`. Returns the array.
6. After hydration, `usePreloadedQuery` keeps the subscription live so owner edits in another tab propagate immediately.

**Write (owner-only):**
1. Owner clicks Add Entry → `<BioEntryFormDialog>` opens with empty defaults.
2. Owner submits the form. `useMutation(api.bio.mutations.create)` is called with `{ kind, title, startDate, endDate, description?, link? }`. **No `userId` arg.**
3. Server: `authMutation` resolves `ctx.user`; `getAppUser(ctx, ctx.user._id)` → `appUserId`. The new row is inserted with `userId: appUserId`. Soft-cap check: count entries; throw if ≥ 50.
4. Same handler schedules `internal.embeddings.actions.generateEmbedding({ sourceTable: "bioEntries", sourceId: <newId> })` via `ctx.scheduler.runAfter(0, ...)`.
5. The scheduled action calls `internal.embeddings.queries.getContentForEmbedding`, which loads the new row and serializes it as **natural prose** via `serialize-for-embedding.ts` (issues I6, iter2-Finding6). Use **subject-neutral phrasing** that avoids first/third-person pronouns to keep the chunk natural under either system-prompt voicing. Example for a Work entry: `"Worked as Senior Engineer at Acme from January 2022 to March 2024. Led the migration of the billing pipeline. More info: https://acme.com/blog/billing"`. Education example: `"Studied Computer Science at MIT from September 2014 to May 2018."`. The action then takes the `kind === "bio"` branch (no published-status check), embeds the prose as a single chunk, and inserts into `contentEmbeddings` with `userId = appUserId`, `sourceTable = "bioEntries"`, `sourceId`, `chunkIndex = 0`, `slug = undefined`.
6. UI: Convex live query reflects the new row immediately; the new entry card appears at top of the list.

**Update:**
- Same as create, but uses `ctx.db.patch` after `isOwnedByUser` check; embedding pipeline is re-triggered, which deletes old embeddings for `(sourceTable, sourceId)` and re-inserts the new chunk. (This is how the existing pipeline already works for articles/posts edits — confirmed by codebase analyst.)

**Remove:**
- After `isOwnedByUser` check, `ctx.db.delete(id)` and schedule `internal.embeddings.mutations.deleteBySource({ sourceTable: "bioEntries", sourceId })`.

**Chat retrieval (visitor talks to clone):**
1. Visitor sends a message in `/@username/chat`.
2. `chat/actions.ts:streamResponse` runs vector search: `ctx.vectorSearch("contentEmbeddings", "by_embedding", { vector: queryEmbedding, filter: q.eq("userId", profileOwnerId), limit: 6 })`.
3. The filter on `userId` is the **single** isolation barrier. Because bio entries are inserted with `userId = profileOwnerId` (server-derived), they participate naturally; rows belonging to other users are filtered out by the same call site that already protects articles/posts.
4. Retrieved chunks (mix of bio + articles + posts, ranked by similarity) are joined into the system prompt under the renamed header "Relevant Background and Writing".

### 3. Why this works

**Invariants preserved:**
- The chat retrieval call site (`chat/actions.ts`) already filters by `userId`; bio entries inherit this guarantee for free. **No new isolation surface is introduced.**
- The embedding write path always sets `userId` from a server-derived value via `getAppUser(ctx.user._id)`. Client cannot supply or override it.
- The existing `(sourceTable, sourceId)` pair stays the unique handle for an embedded record across the whole system; bio adopts it without extension.
- The schema change is strictly additive: a new table + one new literal in a union. No existing function, validator, or test is rewritten — only references to the validator may change as part of the shared-validator extraction.

**Bug classes made impossible (or much harder):**
- _Cross-user RAG leak from bio_: the chat retrieval filter already pins to `userId`. Combined with server-derived write `userId`, no client request can plant or surface another user's entries.
- _Stale embeddings after edit/delete_: the pipeline re-deletes by `(sourceTable, sourceId)` on update before re-inserting; delete schedules `deleteBySource`. Mirroring the existing articles/posts contract avoids divergence.
- _Tab broken after rename_: `PROFILE_TAB_KINDS` is the single source of truth; `isProfileTabKind`, `getProfileTabHref`, `PROFILE_TAB_LABELS` all flow from that tuple. Adding `"bio"` in one place wires the whole route guard chain.
- _Forgotten test coverage_: the `__tests__/types.test.ts` file already enumerates kinds; updating it forces a compile-time reminder to keep the union and the test in sync.

**Existing behavior unchanged:**
- Articles/posts ingestion path: unchanged except that the validator for `sourceTable` is extracted to a shared symbol. We will add a Vitest assertion that the existing articles/posts embedding flow round-trips identically before and after the validator extraction (one snapshot test).
- Chat system prompt composition: the only change is the header string. We will not modify `composeSystemPrompt` token budget logic, system-prompt ordering, or any other helper.
- Other tabs (`posts`, `articles`, `clone-settings`): the only file they share with bio is `profile-tabs/types.ts`. The new `"bio"` kind is appended at index 2 in `PROFILE_TAB_KINDS` (not the front), and `PROFILE_TAB_DEFAULT_KIND` is changed to a literal `"posts"` rather than `PROFILE_TAB_KINDS[0]` — so the workspace's default-fallback-tab semantics are unchanged. Visual ordering moves to a separate `PROFILE_TAB_DISPLAY_ORDER` constant.

**Alternatives considered**

| Option | Tradeoff | Rejected because |
| --- | --- | --- |
| Inject all bio entries into the system prompt directly (no embeddings) | Cheaper for tiny corpora; one fewer scheduled action; no vector search cost. **But:** at the 50-entry cap, full-serialized text could exceed the existing `SYSTEM_PROMPT_MAX_CHARS = 6000` budget when stacked with persona prompt + tone presets + topics-to-avoid, forcing truncation. Also splits the agent's "knowledge" into two retrieval modes. | The user explicitly chose RAG (clarification #5), and architectural uniformity with articles/posts ingestion compounds: one path, one set of tests, one filter. |
| Single text blob field on the `users` table (treat bio as an unstructured biography blob) | Simplest schema change. | Loses structured display (cards, sortable date ranges, optional link), loses per-entry edit/delete, loses kind filter for future UI. The user explicitly asked for structured fields. |
| Separate `workEntries` and `educationEntries` tables | Cleanly typed by domain. | The user chose a single shape with a `kind` discriminator (clarification #4). Two tables would double schema, mutations, embedding branches, and tests for no caller-visible win. |
| Store dates as `{ year, month }` object | More explicit than epoch ms; no UTC drift risk for "month" semantics. | `v.number()` sorts and indexes natively in Convex; epoch ms anchored to UTC first-of-month yields correct month/year on display via `format-date-range.ts`. The number-as-month-anchor approach is one validator instead of two and matches existing date storage in the repo. |
| Skip the soft cap (50 entries) | Simpler validator. | Without it, an attacker (or a user pasting a long career) could blow up vector index size or system prompt token budget when retrieved chunks are concatenated. 50 is generous for genuine human history. |
| Drag-to-reorder with manual `position` integers | Editorial control. | Adds a position column, drag UX, reorder mutation, and migration risk. No proven user need at this corpus size. Auto-sort by `startDate` desc satisfies the user's stated requirement. |

### 4. Edge cases and gotchas

- **`startDate` only — no end date.** Means "Present" (current role/study). UI shows "Jan 2022 – Present"; sort uses `startDate` only (the entry stays at the top until a newer entry with a later start date is added). Validation: if `endDate != null`, must be ≥ `startDate`.
- **Same-month start and end.** Allowed; renders as "Jan 2024" (single month string when `start === end`).
- **Future start dates.** Allow up to `currentYear + 1` so users can announce upcoming roles. Reject anything beyond.
- **Pre-1900 dates.** Rejected by Zod (range `1900 ≤ year ≤ currentYear + 1`). Catches form-input mistakes (e.g. typing 0024 instead of 2024).
- **Unicode in title/description.** Allowed. Convex strings are UTF-8 up to 1 MB; we cap at 200/500 chars in the schema, so the byte ceiling is irrelevant in practice.
- **URL link.** Either empty string or a valid `https://` URL (Zod `.url()`). On render, `target="_blank"` and `rel="noopener noreferrer"` to prevent reverse-tabnabbing. Stored as-is; no auto-prefixing of `https://` (display layer is honest about what the user typed).
- **Entry deletion race.** If a delete and an update fire near-simultaneously, the existing scheduler ordering still resolves correctly: each is a separate scheduled action keyed by `(sourceTable, sourceId)`. The last write wins on the embedding row; if delete runs after the re-embed completes, the embedding row is removed cleanly.
- **Soft-cap of 50 is best-effort, not transactional (issues I7, iter2-Finding5).** The `create` mutation does a count-then-insert. In a single-process call chain Convex serializes mutations, but two separate Convex worker processes can both observe `count === 49` before either commits → 51 entries. This is benign (no security impact, no data loss) and self-heals on the next `create` rejection. The Vitest only verifies the in-process sequential behavior — the cross-process race is a documented untested edge.
- **Panel state for `routeState=null` tabs (issue C3).** `bio` adopts the same shape as `clone-settings`: `hasContentRoute=true` (because `isProfileTabKind("bio")` returns true) and `routeState=null` (carved out in `getContentRouteState` via the inverted predicate). The workspace-shell already handles this state for `clone-settings`; the Playwright test for FR-01 explicitly asserts the content panel is **open** (not collapsed) when navigating to `/@username/bio`, to lock the contract.
- **Empty-state for owner.** Shows "Tell readers a bit about your background" with an Add CTA. For visitors, shows "No entries yet."
- **`bio` short field on `users` table is unrelated.** The existing `bio` string column on `users` is a one-line tagline ("Designer in NYC"). It stays. Naming overlap is unfortunate; the codebase analyst confirmed it. We document this in the file header of `convex/bio/schema.ts` to prevent future confusion. If we want to rename the legacy field later, that's a separate spec.
- **Embedding pipeline cost.** Bio entries are short (typical < 500 chars); chunker is bypassed (single chunk per entry). At 50 entries, that's at most 50 embeddings per user — well under any current limits.
- **Backfill.** Not applicable for v1 — the table is brand new and no existing user has data. Existing `internal.embeddings.actions.backfillEmbeddings` covers the future case if a migration adds entries en masse.
- **Hidden tab on signed-out users (regression risk).** Make sure the new `"bio"` kind is **not** filtered out alongside `"clone-settings"` in `profile-tabs.tsx` (which currently filters `clone-settings` for non-owners). The filter must remain `kind === "clone-settings"` only — adding `bio` to the kinds tuple should not accidentally hide it.
- **`@content/default.tsx` returns null.** Confirmed it falls through to `bio/page.tsx` correctly because Next's parallel-route resolver matches on directory presence first.
- **SSR-to-client hydration flash (issue I8).** Without preloading, a server-rendered list would flash to a `useQuery` loading skeleton before the Convex client populates. The `@content/bio/layout.tsx` calls `preloadAuthOptionalQuery` and the panel consumes it via `usePreloadedQuery` — no flash. This mirrors the articles pattern. Skipping the layout would inherit the flash.
- **Status gate in `generateEmbedding` (issue C1).** The current implementation aborts (and clears existing chunks) when `content.status !== "published"`. Bio entries do not have a `status` field. The fix is to discriminate on `content.kind` ("doc" vs "bio"), not to fabricate a `status: "published"` constant in the bio branch — the latter is a sentinel-disguised lie that breaks the moment a real `status` field is added to bio. The Vitest in FR-12 explicitly asserts that bio embeddings are **not** deleted by the gate.
